// 04 — Vulnerabilidades intencionadas

Diseñadas para el pentesting

⚠️ IMPORTANTE: Estas vulnerabilidades son intencionadas y documentadas. El sistema debe desplegarse en un entorno aislado y controlado, nunca expuesto a internet real sin advertencias claras. Siempre con consentimiento explícito de todos los participantes.
CRÍTICA
SQL Injection en login
El endpoint POST /api/auth/login construye la query SQL concatenando strings directamente. ' OR '1'='1 permite bypass de autenticación.
CRÍTICA
JWT Secret débil
El secret para firmar tokens JWT es "secret123". Con hashcat o jwt_tool se puede forjar tokens de admin en minutos.
ALTA
IDOR en documentos
El endpoint GET /api/docs/{id} no verifica que el documento pertenezca al usuario autenticado. Cambiar el ID permite leer docs de otras empresas.
ALTA
File Upload sin validación
El endpoint de subida de documentos acepta cualquier tipo de archivo, incluyendo .php o scripts. No hay validación de Content-Type ni magic bytes.
MEDIA
XSS en nombre de empresa
El campo "nombre de empresa" se renderiza sin sanitizar en el dashboard. Almacenar <script>alert(1)</script> ejecuta JS en todos los usuarios que lo vean.
MEDIA
CORS wildcard
La API acepta peticiones de cualquier origen (Access-Control-Allow-Origin: *). Permite ataques CSRF desde dominios maliciosos.
MEDIA
Verbose Error Messages
Los errores devuelven stack traces completos con rutas del servidor, versiones de librerías y estructura de la base de datos. Information disclosure crítico.
BAJA
Rate Limiting ausente
No hay límite en intentos de login. Permite fuerza bruta de credenciales sin restricción temporal ni bloqueo de cuenta.
BAJA
Sensitive Data in Logs
Los logs de acceso incluyen passwords en texto plano y tokens JWT completos. Acceso a /logs o a los contenedores Docker revela credenciales.
💡 Para la sesión de pentesting: Cread un documento "Scope & Rules of Engagement" que defina qué está en scope, qué no (ej: no DoS real), y cómo reportar hallazgos. Al finalizar, haced un CTF-style writeup colectivo. Esto es experiencia real de red team.

// 06 — Cómo usar Claude Code en este proyecto

Guía de herramientas con IA

01

Instalar Claude Code y configurarlo en el repo

Claude Code es un agente de terminal. En cada módulo (backend/, frontend/, nlp-engine/) podéis invocar Claude Code para que escriba código, corrija bugs y explique la arquitectura. Es especialmente útil para el DevOps: puede escribir Dockerfiles completos, configurar nginx, y generar el docker-compose.yml entero con un solo prompt bien estructurado.

02

Generar el scaffold inicial del backend

Desde el directorio backend/, pedid a Claude Code: "Crea una aplicación FastAPI con autenticación JWT, modelos SQLAlchemy para User, Company, Document y NormativaResult, y endpoints CRUD básicos. Usa PostgreSQL. Incluye Alembic para migraciones." — En minutos tendréis una base funcional que el backend lead puede refinar.

03

Motor NLP: iteración rápida con Claude

El NLP engineer puede usar Claude (aquí, en el chat) para diseñar el pipeline primero, luego usar Claude Code para implementarlo. Prompt sugerido: "Dado un texto de política de empresa y una lista de requisitos de ISO 27001, crea una función Python que devuelva qué requisitos están cubiertos, cuáles parcialmente y cuáles ausentes, usando sentence-transformers para comparación semántica."

04

Frontend: componentes con diseño real

El frontend developer puede usar Claude (chat) para generar componentes React completos. Sed específicos: describid el estado, las props, y el comportamiento esperado. Claude Code puede luego integrar esos componentes en el proyecto real, resolver imports y ajustar estilos.

05

Diseñar vulnerabilidades con explicación pedagógica

La persona de Security puede usar Claude para: 1) Obtener el código exacto de una SQLi vulnerable en FastAPI, 2) Generar la guía de pentesting paso a paso para los compañeros, y 3) Escribir los writeups post-explotación. Esto garantiza que las vulns estén bien implementadas y sean reproducibles.

06

Code Review automático con Claude

Antes de cada PR, pasad el diff por Claude: "Revisa este código Python buscando: bugs lógicos, problemas de seguridad no intencionados, y mejoras de rendimiento." Esto evita que introduzcáis vulnerabilidades accidentales además de las diseñadas, y mejora la calidad del código general.

07

Documentación automática

Al final de cada semana, usad Claude para generar documentación: CHANGELOG, actualizar el README, generar ejemplos de uso de la API. Claude Code puede leer vuestros archivos directamente y producir documentación que refleje el estado real del código.
